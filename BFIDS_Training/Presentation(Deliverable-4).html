User Profiling: AVL Trees NON-LINEAR DATA STRUCTURE

Concept: Self-Balancing BST

We use AVL Trees to store user process profiles (e.g., "Chrome", "Word").

Why Apply This?

The Problem: User data often arrives in sorted order (e.g., timestamps). A standard BST would degrade into a Linked List (Skewed Tree), resulting in O(n) search time.

The Fix: AVL Trees enforce balance. This guarantees that checking "Is this process normal?" always takes O(log n) time, crucial for low-latency detection.

Workflow Analysis: Graphs GRAPH THEORY

Concept: Directed Graph

We use Adjacency Lists to model user workflow. Nodes are apps; edges are transitions (IDE → Terminal).

Why Apply This?

  • BFS (Breadth-First): Used to validate if a sequence of user actions follows the shortest/most logical known path.
  • DFS (Depth-First): Essential for detecting Cycles. Automated bots often get stuck in loops (A→B→A); humans rarely do.
Alert System: Max-Heap PRIORITY QUEUES

Concept: Priority Queue

We use a Max-Heap to manage real-time security alerts.

Why Apply This?

The Problem: In a Security Operations Center (SOC), thousands of logs occur. We cannot simply sort the array every time (O(n log n)) to find the worst threat.

The Fix: A Max-Heap keeps the highest "Anomaly Score" at the root at all times. This gives us O(1) instant access to the most critical threat.

Pattern Matching: Trie STRING ALGORITHMS

Concept: Prefix Tree

We use a Trie to implement the security blacklist (blocking cmd.exe, cryptolocker.exe).

Why Apply This?

The Problem: Hash tables only match exact strings. They are slow at checking "Does this start with 'crypto'?"

The Fix: Tries perform prefix matching in O(L) time, where L is the word length. Crucially, the speed is independent of the database size. Blocking 1 million viruses is as fast as blocking 1.

Scalability: B-Tree DISK OPTIMIZATION

Concept: Multi-way Search Tree

We use B-Trees to index the persistent storage of user fingerprints.

Why Apply This?

The Problem: AVL Trees are optimized for RAM (Memory). When data grows to Terabytes, it moves to Disk. Disk access is 1,000x slower than RAM.

The Fix: B-Trees are "fat" trees with hundreds of keys per node. This structure matches the size of a Disk Page (4KB), minimizing the number of slow I/O operations required to fetch data.

Algorithmic Core ALGORITHMS

Sorting

Quick Sort: Recursive, In-Place. Used for initial high-speed log ingestion.

Merge Sort: Stable. Used when relative order of timestamps matters.

Searching

Binary Search: O(log n) lookup on sorted arrays to find specific log entries.

Trie Search: O(L) prefix lookup for instant blacklisting.

Traversal

Recursion: Core to our DFS, Tree Rotations, and Sorting logic.

Hashing: Custom Hash Map for O(1) User ID → Profile mapping.
